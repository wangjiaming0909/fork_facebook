1 ## Proxygen: Facebook's C++ HTTP Libraries
2 
3 [![Build Status](https://travis-ci.org/facebook/proxygen.svg?branch=master)](https://travis-ci.org/facebook/proxygen)
4 
5 This project comprises the core C++ HTTP abstractions used at
6 Facebook. Internally, it is used as the basis for building many HTTP
7 servers, proxies, and clients. This release focuses on the common HTTP
8 abstractions and our simple HTTPServer framework. Future releases will
9 provide simple client APIs as well. The framework supports HTTP/1.1,
10 SPDY/3, SPDY/3.1, and HTTP/2. The goal is to provide a simple,
11 performant, and modern C++ HTTP library.
12 
13 We have a Google group for general discussions at https://groups.google.com/d/forum/facebook-proxygen.
14 
15 The [original blog post](https://code.facebook.com/posts/1503205539947302)
16 also has more background on the project.

18 ### Installing
19 
20 Note that currently this project has only been tested on Ubuntu 14.04,
21 although it likely works on many other platforms. Support for Mac OSX is
22 incomplete.
23 
24 You will need at least 3 GiB of memory to compile `proxygen` and its
25 dependencies.
26 
27 ##### Easy Install
28 
29 Just run `./deps.sh` from the `proxygen/` directory to get and build all
30 the dependencies and `proxygen`. It will also run all the tests. Then run
31 `./reinstall.sh` to install it. You can run `./deps.sh && ./reinstall.sh`
32 whenever to rebase the dependencies, and then rebuild and reinstall `proxygen`.
33 
34 A note on compatibility: this project relies on system installed
35 [folly](https://github.com/facebook/folly). If you rebase `proxygen` and `make` starts to fail, you likely
36 need to update to the latest version of `folly`. Running
37 `./deps.sh && ./reinstall.sh` will do this for you. We are still working
38 on a solution to manage dependencies more predictably.
39 
40 ##### Other Platforms
41 
42 If you are running on another platform, you may need to install several
43 packages first. Proxygen and `folly` are all autotools based projects.

58 ### Architecture
59 
60 The central abstractions to understand in `proxygen/lib` are the session, codec,
61 transaction, and handler. These are the lowest level abstractions, and we
62 don't generally recommend building off of these directly.
63 
64 When bytes are read off the wire, the `HTTPCodec` stored inside
65 `HTTPSession` parses these into higher level objects and associates with
66 it a transaction identifier. The codec then calls into `HTTPSession` which
67 is responsible for maintaining the mapping between transaction identifier
68 and `HTTPTransaction` objects. Each HTTP request/response pair has a
69 separate `HTTPTransaction` object. Finally, `HTTPTransaction` forwards the
70 call to a handler object which implements `HTTPTransaction::Handler`. The
71 handler is responsible for implementing business logic for the request or
72 response.
73 
74 The handler then calls back into the transaction to generate egress
75 (whether the egress is a request or response). The call flows from the
76 transaction back to the session, which uses the codec to convert the
77 higher level semantics of the particular call into the appropriate bytes
78 to send on the wire.
79 
80 The same handler and transaction interfaces are used to both create requests
81 and handle responses. The API is generic enough to allow
82 both. `HTTPSession` is specialized slightly differently depending on
83 whether you are using the connection to issue or respond to HTTP
84 requests.
85 
86 ![Core Proxygen Architecture](CoreProxygenArchitecture.png)
87 
88 Moving into higher levels of abstraction, `proxygen/httpserver` has a
89 simpler set of APIs and is the recommended way to interface with `proxygen`
90 when acting as a server if you don't need the full control of the lower
91 level abstractions.
92 
93 The basic components here are `HTTPServer`, `RequestHandlerFactory`, and
94 `RequestHandler`. An `HTTPServer` takes some configuration and is given a
95 `RequestHandlerFactory`. Once the server is started, the installed
96 `RequestHandlerFactory` spawns a `RequestHandler` for each HTTP
97 request. `RequestHandler` is a simple interface users of the library
98 implement. Subclasses of `RequestHandler` should use the inherited
99 protected member `ResponseHandler* downstream_` to send the response.

101 ### Using it
102 
103 Proxygen is a library. After installing it, you can build your own C++
104 server. Try `cd`ing to the directory containing the echo server at
105 `proxygen/httpserver/samples/echo/`. You can then build it with this one
106 liner:
107 
108 <code>
109 g++ -std=c++14 -o my_echo EchoServer.cpp EchoHandler.cpp -lproxygenhttpserver -lfolly -lglog -lgflags -pthread
110 </code>
111 
112 After running `./my_echo`, we can verify it works using curl in a different terminal:
113 ```shell
114 $ curl -v http://localhost:11000/
115 * Trying 127.0.0.1...

117 > GET / HTTP/1.1
118 > User-Agent: curl/7.35.0
119 > Host: localhost:11000
120 > Accept: */*
121 >
122 < HTTP/1.1 200 OK
123 < Request-Number: 1
124 < Date: Thu, 30 Oct 2014 17:07:36 GMT
125 < Connection: keep-alive
126 < Content-Length: 0
127 <
128 * Connection #0 to host localhost left intact
